SEÇÃO 03

Diretrizes & padrões visuais.

As regras do jogo antes de colocar a mão na massa. Esta seção define como consumir o Design System, nomear camadas e organizar arquivos — três pilares que mantêm nossos projetos legíveis para o time todo.

01 / Design System

Como consumir nossas bibliotecas globais.

Toda interface começa pelo Design System. Ele é a fonte única da verdade — cores, espaçamentos, tipografia e componentes vivem lá.

  • Ative a biblioteca OL/Foundations e OL/Components em todo arquivo novo.
  • Nunca faça override de componentes — proponha uma evolução no DS.
  • Use sempre tokens: cor, espaçamento e tipografia por variável, nunca hard-coded.
  • Encontrou um gap? Abra uma request no canal do DS.
02 / Nomenclatura

Camadas e componentes legíveis.

Uma camada bem nomeada é um presente para o próximo designer (e para o dev). Seja descritivo, evite genéricos como Rectangle 23.

  • Padrão: função/elemento/estado — ex.: btn/primary/hover.
  • Use kebab-case e remova números automáticos do Figma.
  • Variants seguem props documentadas (size, state, variant).
  • Tokens com nome semântico, nunca cromático: bg-primary, não white.
03 / Estrutura

Organização de páginas no arquivo.

Todo arquivo do time segue a mesma anatomia. Isso torna a navegação previsível mesmo para quem entra no projeto pela primeira vez.

  • Cover — capa com contexto do projeto.
  • 💻 Protótipo — telas em alta fidelidade, prontas para handoff.
  • 🎨 Componentes — instâncias locais e exceções do DS.
  • 📦 Arquivo — versões antigas e explorações.

Estrutura padrão de um arquivo.

Toda nova file segue exatamente esta sequência de páginas — independente do produto, time ou tamanho do projeto.

  • 📌 Cover contexto · status · owner
  • 💻 Protótipo telas finais
  • 🎨 Componentes instâncias locais
  • 🧪 Explorações drafts & testes
  • 📦 Arquivo versões antigas
Não faça isso

Nada de cores soltas, frames sem nome ou componentes “destacados”.

Destacar (detach) um componente do Design System quebra a propagação de melhorias e polui a biblioteca. Se precisa de algo diferente, abra uma proposta — não destaque. O mesmo vale para usar HEX direto na propriedade em vez do token.

Atenção

Telas “órfãs” não passam no review.

Toda tela precisa estar dentro de uma seção nomeada e conectada a pelo menos um fluxo. Telas soltas no canvas são removidas no review semanal do DS.

05 · FAQ rápido

Dúvidas? A gente responde.

Posso desanexar (detach) um componente do Design System?
Não. Detach quebra a propagação de updates e polui o arquivo. Se você precisa de algo que não existe, abra uma proposta no canal do DS — quase sempre é melhor evoluir o componente do que criar uma cópia local.
Encontrei um bug ou inconsistência no Design System. E agora?
Reporte direto no canal #design-system com print, link do arquivo e descrição do comportamento esperado. Triagem é semanal, mas issues críticas entram em pauta no mesmo dia.
Posso usar fontes externas em um projeto pontual?
Apenas em peças de marketing aprovadas. Em produto, sempre as fontes do DS — elas estão otimizadas para legibilidade e performance.
Como sugiro uma evolução para o Design System?
Use o template DS Request no Figma e marque o time no canal. Toda proposta passa por revisão técnica e visual antes de virar release.
Existe versionamento do Design System?
Sim. Seguimos semver (major.minor.patch) e publicamos changelog quinzenal. Mudanças breaking são comunicadas com 14 dias de antecedência.
